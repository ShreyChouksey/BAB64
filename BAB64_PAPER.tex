\begin{abstract}
We present BAB64, a proof-of-work construction in which the hash function is not fixed at protocol design time but is instead derived from the input itself. Each candidate input deterministically generates a 64$\times$64 grayscale image, which in turn parameterizes a unique substitution-permutation network---defining its own round constants, rotation schedule, S-box, and initial state. The proof-of-work output is the image hashed by its own derived function. This self-referential structure eliminates precomputation advantages and creates structural ASIC resistance by requiring per-nonce function reconfiguration.

Empirical evaluation over 101 tests shows near-ideal avalanche (50.2\%), uniform bit distribution ($P(1) = 0.4998$), zero collisions across 500 trials, and byte-identical outputs across two independent implementations. A parallel diffusion step achieves 46.3\% single-block avalanche in one round. Six adversarial attack simulations---including related-image, preimage structure, and Joux multi-collision---find no exploitable weakness. S-box quality analysis across 10{,}000 images shows mean nonlinearity of 91.8, consistent with random permutations. A C-accelerated implementation achieves approximately 7{,}000 hashes per second on commodity hardware.

An identity layer provides Bitcoin-like addresses derived from private images, with Lamport one-time signatures for quantum-resistant transaction signing, automatic key rotation, and replay protection.

No formal security reduction is provided; the construction's resistance to optimization rests on structural arguments and empirical evidence.
\end{abstract}

\section{Introduction}

Proof-of-work (PoW) systems underpin the security of permissionless consensus protocols. In all widely deployed PoW constructions---Bitcoin's SHA-256d~\cite{nakamoto2008}, Ethereum's Ethash~\cite{buterin2014}, Zcash's Equihash~\cite{biryukov2016}, and Cuckoo Cycle~\cite{tromp2015}---the hash function is fixed at protocol design time. The miner searches for an input that produces a low output under this fixed function. This architectural choice has two consequences:

\begin{enumerate}
    \item \textbf{Precomputation is possible.} A fixed function admits partial evaluation reuse. Bitcoin miners exploit midstate caching; Ethash requires a multi-gigabyte DAG precomputed per epoch.

    \item \textbf{ASIC optimization is straightforward.} When the function is known at fabrication time, its constants, rotation amounts, and data paths can be hardcoded into silicon. Bitcoin ASICs achieve orders-of-magnitude efficiency over general-purpose hardware precisely because SHA-256 never changes.
\end{enumerate}

BAB64 eliminates the fixed function entirely. Each mining attempt produces a candidate image $I$, which deterministically defines a unique compression function $H_I$. The proof-of-work output is $H_I(I)$---the image hashed by the function it defines. The miner searches simultaneously in input space and function space.

This paper makes four contributions:

\begin{itemize}
    \item A \textbf{novel PoW primitive} based on self-referential hashing, where no two mining attempts use the same hash function.
    \item A \textbf{concrete construction} with byte-precise specification, two independent implementations, and 101 passing tests across core hashing, identity, and stress test suites.
    \item \textbf{Empirical evidence} that the construction achieves near-ideal statistical properties despite using data-derived, per-input parameters, validated by six adversarial attack simulations.
    \item An \textbf{identity layer} with private-image-derived addresses and quantum-resistant Lamport signatures, demonstrating that BAB64 can serve as a foundation for a complete transaction system.
\end{itemize}

We emphasize that BAB64 is a research contribution. No formal security reduction is provided, and the construction has not been deployed in an adversarial setting. We present the design, evidence, and limitations honestly and invite further analysis.

\section{Construction}

\subsection{Design Intuition}

The construction draws on a thought experiment from Borges' \emph{Library of Babel}: a combinatorial library containing every possible book. BAB64 treats the space of 64$\times$64 grayscale images as such a library, where each ``book'' contains instructions for its own cryptographic compression. The miner's task is to find a book whose self-description meets an external difficulty criterion.

Concretely, each image provides the following hash function components:

\begin{table}[h]
\centering
\small
\begin{tabular}{@{}lp{4.2cm}l@{}}
\toprule
\textbf{Component} & \textbf{Derivation} & \textbf{Analogue} \\
\midrule
$K[0..31]$ & SHA-256 of 128-pixel blocks & SHA-256's $K[]$ \\
$\mathit{rot}[0..31]$ & Pixel pair lookup, $[1, 31]$ & SHA-256's ROTR amounts \\
$\mathit{sbox}[0..255]$ & Fisher--Yates from image hash & AES Rijndael S-box \\
$H_0[0..7]$ & SHA-256 of full image & SHA-256's $H_0$ \\
\bottomrule
\end{tabular}
\caption{Hash function components derived from each image.}
\end{table}

The derived function follows Merkle--Damg{\aa}rd structure with Davies--Meyer feedforward, processing the image as 128 blocks of 32 bytes through 32 rounds of substitution-permutation per block.

\subsection{Self-Referential Structure}

The defining property of BAB64 is that the hash function's parameters are derived from the same data being hashed. This creates a circular dependency that is resolved by separating parameter derivation (which uses SHA-256 on image regions) from hash evaluation (which uses the derived SPN on the full image). The image is both the message and the key schedule.

This structure ensures that no two nonces share hash function parameters, since the image changes with every nonce, and the image fully determines the function.

\section{Algorithm}

\subsection{Image Generation}

Given input string \texttt{input\_data} and integer nonce:

\begin{lstlisting}
base_seed  = SHA-256(input_data)
image_seed = SHA-256(base_seed || nonce)     // 8-byte big-endian nonce
image      = CSPRNG_expand(image_seed, 4096) // SHA-256 in counter mode
\end{lstlisting}

The expansion requires $\lceil 4096 / 32 \rceil = 128$ SHA-256 evaluations. Each seed produces exactly one 4{,}096-byte image.

\subsection{Parameter Derivation}

\textbf{Round constants.} The image is partitioned into 32 blocks of 128 pixels. Each block is SHA-256 hashed; the first 4 bytes of the digest become a 32-bit round constant.

\textbf{Rotation schedule.} For each round $r$, two pixels at offset $N/2 + 2r$ are combined: $\mathit{rot}[r] = ((p_1 \times 256 + p_2) \bmod 31) + 1$, guaranteeing the range $[1, 31]$.

\textbf{S-box.} A Fisher--Yates shuffle of $[0..255]$ seeded by $\text{SHA-256}(\text{image} \| \texttt{b'sbox'})$, producing a random permutation.

\textbf{Initial state.} Eight 32-bit words from $\text{SHA-256}(\text{image} \| \texttt{b'init\_state'})$.

\subsection{Compression Function}

The compression function takes a 256-bit state and 256-bit message block, producing a 256-bit state. Each of 32 rounds applies:

\begin{enumerate}
    \item \textbf{S-box substitution} on $s[0]$ (4 bytes through $\mathit{sbox}[]$)---non-linearity
    \item \textbf{Bit rotation} $\text{ROTR32}(s[0], \mathit{rot}[r])$---diffusion
    \item \textbf{XOR} with round constant $K[r]$ and message word $w[r \bmod 8]$---key mixing
    \item \textbf{Modular addition} $s[0] = (s[0] + s[1]) \bmod 2^{32}$---non-linear mixing
    \item \textbf{Multi-word injection} $w[r]$ added to $s[4]$---both state halves touched
    \item \textbf{Majority function} $\text{maj}(s[0], s[1], s[2])$ added to $s[3]$---cross-word coupling
    \item \textbf{Choice function} $\text{ch}(s[4], s[5], s[6])$ added to $s[7]$---conditional mixing
    \item \textbf{Word rotation} of the state array---positional diffusion
    \item \textbf{Parallel diffusion} (MixColumns-like)---each word XORs rotated bits from two non-adjacent words
\end{enumerate}

Step~9 is critical: for each word $i$, $s[i] \mathrel{\oplus}= \text{ROTR32}(s[(i{+}2) \bmod 8], 11) \oplus \text{ROTR32}(s[(i{+}5) \bmod 8], 19)$. This ensures a change in \emph{any} word propagates to all 8 words within a single round.

After 32 rounds, Davies--Meyer feedforward adds the input state to the output, preventing invertibility.

\subsection{Full Hash}

Merkle--Damg{\aa}rd iteration over 128 blocks of 32 bytes. Total work: $128 \times 32 = 4{,}096$ round function evaluations per hash.

\subsection{Proof and Verification}

A proof consists of $(\texttt{input\_data}, \texttt{base\_seed}, \texttt{nonce}, \texttt{image\_hash}, \texttt{bab64\_hash}, \texttt{difficulty})$. Verification regenerates the image, recomputes the self-referential hash, and checks seed consistency, image integrity, hash correctness, and difficulty. Verification cost equals exactly one mining attempt.

\section{Security Analysis}

\subsection{Threat Model}

The adversary controls the mining software, may choose any nonce, implement arbitrary solving strategies, and use specialized hardware. The adversary cannot modify the verification algorithm or predict SHA-256 outputs.

\subsection{Resistance to Weak-Function Targeting}

Three structural properties prevent an attacker from finding images that produce degenerate hash functions:

\textbf{Parameter space.} The S-box alone admits $256! \approx 2^{1684}$ possible values.

\textbf{SHA-256 indirection.} Images are SHA-256-expanded from nonces, not freely chosen. Targeting specific pixel patterns requires inverting SHA-256.

\textbf{Structural minima.} The S-box is always a valid permutation, rotations are always in $[1, 31]$, and round constants are SHA-256 digests.

\subsection{Preimage and Precomputation Resistance}

Finding $I$ such that $H_I(I)$ has $d$ leading zero bits requires the full evaluation pipeline. Since the hash function changes with every nonce, no precomputation carries across attempts. Each trial succeeds independently with probability $2^{-d}$.

\subsection{ASIC Resistance}

A BAB64 ASIC would need to load a new S-box (256~B), round constants (128~B), rotation schedule (128~B), and initial state (32~B) per nonce. This is resistance in degree, not in kind---a dedicated ASIC could still outperform a CPU, but the efficiency ratio would be substantially smaller than Bitcoin's.

\subsection{Round Isolation and Parallel Diffusion}

A critical design concern for any SPN is how quickly a single-bit change propagates to all output bits. In the original design (steps 1--8 only), the round function achieved only ${\sim}20.9\%$ single-block avalanche, because changes propagated by only one word position per round.

The parallel diffusion step (step~9) was introduced to address this. After each round's word rotation, every state word absorbs rotated bits from two non-adjacent words. The measured impact:

\begin{table}[h]
\centering
\begin{tabular}{@{}lcc@{}}
\toprule
\textbf{Round} & \textbf{Without parallel diffusion} & \textbf{With parallel diffusion} \\
\midrule
1 & 20.9\% & \textbf{46.3\%} \\
2 & ${\sim}35\%$ & ${\sim}49\%$ \\
4 & ${\sim}44\%$ & ${\sim}50\%$ \\
32 & ${\sim}50\%$ & ${\sim}50\%$ \\
\bottomrule
\end{tabular}
\caption{Single-block avalanche before and after parallel diffusion.}
\end{table}

At round~1, all 8 state words show bit changes (vs.\ only 2--3 without parallel diffusion). The 32-round full hash has significant safety margin: even a 4-round variant maintains adequate diffusion.

\subsection{Attack 4: Related-Image Attack}

\textbf{Threat.} If similar images (differing by 1 pixel) produce exploitably related hash functions, an attacker could search near-miss neighbors for faster-than-brute-force mining.

\textbf{Method.} Four tests over 200--2{,}000 image pairs:

\begin{enumerate}
    \item \textbf{Parameter overlap.} S-box changes completely for 1-pixel neighbors (mean overlap $< 0.01$) due to full-image SHA-256 hashing. Round constants share ${\sim}31/32$ (expected: only one 128-pixel block changes).

    \item \textbf{Hash correlation.} Pearson correlation between corresponding bit positions: mean $|r| < 0.05$ across 500 pairs.

    \item \textbf{Near-miss clustering.} Among 2{,}000 nonces, neighbors of near-misses (6--7 leading zeros) score no better than random ($p > 0.05$).

    \item \textbf{Function distance.} Fixed probe image hashed with both functions: ${\sim}128$ bits different (50\%---effectively independent).
\end{enumerate}

\textbf{Result.} All four tests pass. No exploitable related-image weakness.

\subsection{Attack 5: Preimage Structure}

\textbf{Threat.} The self-referential construction creates a circular constraint. Does this reduce the effective search space?

\textbf{Method.} Four tests over 100--10{,}000 images:

\begin{enumerate}
    \item \textbf{Output bias.} Chi-squared test on 10{,}000 hashes: $p > 0.01$, output is uniform.

    \item \textbf{Hash-to-parameter leakage.} Correlation between hash bytes and image statistics (pixel mean, variance, first/last pixel): max $|r| < 0.05$.

    \item \textbf{Fixed-point search.} Mean byte-level fixed points = 0.125, matching random baseline. No permutation relationships.

    \item \textbf{Second-preimage shortcut.} No correlation between parameter similarity and hash distance ($|r| < 0.05$).
\end{enumerate}

\textbf{Result.} All four tests pass. The self-referential construction does not leak exploitable information.

\subsection{Attack 6: Joux Multi-Collision}

\textbf{Threat.} Joux~\cite{joux2004} showed that for Merkle--Damg{\aa}rd hashes with a \emph{fixed} compression function, $2^k$ collisions require only $k$ times the work of one collision. BAB64 uses Merkle--Damg{\aa}rd. Does per-image parameterization neutralize this?

\textbf{Method.} Three tests using the C-accelerated compression:

\begin{enumerate}
    \item \textbf{Internal state collision.} 5{,}000 images, intermediate state at block~64: zero collisions (birthday bound ${\sim}2^{128}$).

    \item \textbf{Block-level collision independence.} In standard MD with a fixed function, block-$N$ collisions propagate to block $N{+}1$. In BAB64, different parameters prevent propagation: average block-1 Hamming distance ${\sim}8.0/8$ words among 100 images.

    \item \textbf{State entropy.} Tracked unique states at blocks 1, 16, 32, 64, 128 across 1{,}000 images: 100\% unique at every checkpoint.
\end{enumerate}

\textbf{Result.} All three tests pass. Per-image parameterization structurally neutralizes Joux's attack by preventing block-level collision composition.

\subsection{Limitations of the Security Argument}

We do not provide a formal reduction to a standard hardness assumption. The security argument rests on structural guarantees, SHA-256 as a random oracle, empirical quality (Section~\ref{sec:empirical}), and six adversarial simulations.

A formal proof of collision resistance for the per-image hash functions remains open. The construction's security is at most as strong as SHA-256.

\section{Identity Layer}

BAB64's self-referential hash creates a natural asymmetry: the private image is expensive to guess but cheap to verify. We exploit this to build a Bitcoin-like identity and transaction system.

\subsection{Identity Construction}

\begin{table}[h]
\centering

\toprule
\textbf{Component} & \textbf{Size} & \textbf{Derivation} \\
\midrule
Private key & 256 bits & Random seed \\
Private image & 4{,}096 bytes & $\text{BabelRender}(\text{private\_key})$ \\
Public address & 256 bits & $H_I(I)$ (self-referential hash) \\
\bottomrule
\end{tabular}

\end{table}

The private image bridges the private key and public address---analogous to a public key that is never revealed. Address uniqueness verified across 1{,}000 identities with uniform first-byte distribution (chi-squared $p > 0.01$).

\subsection{Lamport One-Time Signatures}

Signatures use the Lamport scheme~\cite{lamport1979}, providing quantum resistance by reducing security to SHA-256 preimage hardness. For each bit position $i \in [0, 255]$:

\begin{align*}
sk_0[i] &= \text{SHA-256}(\text{image} \| i \| 0), \quad pk_0[i] = \text{SHA-256}(sk_0[i]) \\
sk_1[i] &= \text{SHA-256}(\text{image} \| i \| 1), \quad pk_1[i] = \text{SHA-256}(sk_1[i])
\end{align*}

To sign message $m$: for each bit $i$ of $\text{SHA-256}(m)$, reveal $sk_0[i]$ or $sk_1[i]$. Verification: hash each revealed secret and compare against the public key. Signature size: 8{,}192 bytes. Verification key: 16{,}384 bytes.

\subsection{Key Rotation and Reuse Prevention}

Lamport signatures are inherently one-time. BAB64Identity maintains a key index counter; each signing operation derives a fresh keypair:

$$\text{keypair}_k = \text{LamportKeyPair}(\text{image\_bytes} \| k)$$

\textbf{Forgery demonstration.} We empirically verified the Lamport reuse vulnerability:

\begin{itemize}
    \item With 2 signatures on the same raw key: ${\sim}50\%$ of positions fully exposed. Selective forgery succeeds.
    \item With 12 reused signatures: ${\sim}256/256$ positions known. Universal forgery succeeds.
    \item With automatic rotation: each signature uses a fresh keypair. Cross-key forgery fails.
\end{itemize}

The raw \texttt{LamportKeyPair} raises \texttt{RuntimeError} on second use, preventing accidental reuse at the API level.

\subsection{Transaction System}

Transactions specify sender, receiver, amount, and a per-transaction nonce. The transaction hash covers all fields; modifying any field invalidates the signature. A \texttt{BAB64TransactionPool} tracks processed hashes for replay protection.

\textbf{End-to-end validation.} 5 identities, 1 mined block, 10 signed transactions, full chain verification, tamper detection, and cross-identity rejection---all pass.

\section{Limitations and Future Work}

\subsection{Limitations}

\textbf{No formal security proof.} The most significant limitation. Six adversarial attack simulations and 101 tests provide empirical confidence but no reduction to a standard hardness assumption.

\textbf{SHA-256 dependency.} Image generation, parameter derivation, Lamport signatures, and verification all use SHA-256. A break of SHA-256 would compromise the entire system.

\textbf{Verification cost.} At ${\sim}0.14$~ms, BAB64 is ${\sim}280\times$ slower to verify than SHA-256d. Acceptable for blockchain block rates but not for sub-microsecond verification.

\textbf{Lamport signature size.} ${\sim}8$~KB per signature with ${\sim}16$~KB verification key---orders of magnitude larger than ECDSA (${\sim}72$~B). A significant cost for quantum resistance.

\textbf{S-box quality below AES.} Mean nonlinearity 91.8 vs.\ AES's 112. Inherent to random permutations vs.\ algebraic optimization. Per-nonce diversity compensates (no fixed attack strategy), but individual instances are weaker.

\textbf{Limited adversarial testing.} No public cryptanalysis, expert differential/algebraic analysis, or adversarial ASIC design analysis.

\subsection{Future Work}

\textbf{Formal analysis.} A proof of collision resistance for the SPN family, even conditional on SHA-256 being a random oracle.

\textbf{Signature compression.} Merkle signature schemes (XMSS, LMS) to compress verification keys. Winternitz OTS to reduce per-signature size.

\textbf{Hardware implementation.} FPGA or ASIC prototype for concrete ASIC resistance data.

\textbf{Extended evaluation.} Full SAC per bit, NIST SP 800-22, Dieharder on BAB64 output streams.

\section{Conclusion}

BAB64 demonstrates that self-referential hashing is a viable foundation for both proof-of-work and identity systems. The construction achieves 50.2\% avalanche, uniform bit distribution, and zero observed collisions, with per-nonce function diversity absent from all deployed PoW systems.

Six adversarial attack simulations confirm resistance to related-image exploitation, preimage structure leakage, and Joux multi-collision composition. The parallel diffusion step achieves 46.3\% avalanche in a single round. S-box quality across 10{,}000 images is consistent with random permutations.

The identity layer demonstrates that BAB64's asymmetry naturally supports Bitcoin-like addresses with quantum-resistant Lamport signatures and automatic key rotation. Forgery demonstrations confirm reuse prevention works.

These properties come at a cost: slower verification (${\sim}0.14$~ms vs.\ ${\sim}0.5~\mu$s), large signatures (${\sim}8$~KB), and structural/empirical rather than formal security. BAB64 is a research prototype, not a deployment-ready system.

We present this work as a contribution to the design space of proof-of-work and identity constructions, backed by 101 tests, six adversarial simulations, and a C-accelerated implementation. We invite the community to evaluate, critique, and extend this construction.
